🔍 GC日志分析平台 - 术语说明测试

📚 术语说明

🏗️ JVM内存结构

堆内存 (Heap)

JVM中存储对象实例的主要内存区域，是垃圾回收的主要目标。包含新生代和老年代。

Eden区

新生代的一部分，新创建的对象首先分配在这里。Eden区满时会触发Minor GC。

Survivor区

新生代的一部分，包含S0和S1两个区域，用于存放经过一次GC后仍存活的对象。

老年代 (Old Generation)

存放长期存活的对象，当对象在新生代中经历多次GC后会被移到老年代。

Metaspace

存储类的元数据信息，如类定义、方法信息等。替代了Java 8之前的永久代。

🔄 GC类型

Minor GC / Young GC

只回收新生代的垃圾回收，频率高但停顿时间短。主要清理Eden区和Survivor区。

Major GC / Old GC

主要回收老年代的垃圾回收，停顿时间较长。通常伴随Minor GC一起执行。

Full GC

回收整个堆内存和Metaspace的垃圾回收，停顿时间最长，应尽量避免频繁发生。

Mixed GC (G1)

G1垃圾回收器特有，同时回收新生代和部分老年代区域的混合回收。

Concurrent GC

与应用程序并发执行的垃圾回收，减少停顿时间，如CMS和G1的并发阶段。

📊 性能指标

停顿时间 (Pause Time)

GC执行期间应用程序暂停的时间，通常以毫秒为单位。停顿时间越短，用户体验越好。

吞吐量 (Throughput)

应用程序运行时间占总时间的比例。吞吐量 = 应用时间 / (应用时间 + GC时间)。

堆利用率

堆内存使用量占堆总容量的百分比。过高可能导致频繁GC，过低则浪费内存。

回收效率

单次GC回收的内存量占GC前内存使用量的比例，反映GC的回收效果。

分配速率

应用程序每秒分配的内存量，影响GC的触发频率。

🎯 垃圾回收器

G1 GC

低延迟垃圾回收器，适合大堆内存应用。将堆分为多个区域，可预测停顿时间。

CMS GC

并发标记清除回收器，主要用于老年代，可与应用程序并发执行。

Parallel GC

并行回收器，使用多线程进行垃圾回收，适合吞吐量优先的应用。

ZGC / Shenandoah

超低延迟垃圾回收器，停顿时间通常在10ms以下，适合对延迟敏感的应用。

IBM J9 GC

IBM JVM的垃圾回收器，包含多种策略如Scavenge、Global GC等。

📈 统计术语

P50 / P90 / P99

百分位数统计。P90表示90%的GC停顿时间都小于等于该值，用于评估性能稳定性。

平均值 (Average)

所有GC事件的算术平均值，提供整体性能的基本概览。

最大值 (Maximum)

观察期间出现的最长停顿时间，反映最坏情况下的性能表现。

标准差

衡量数据分散程度的指标，标准差越小说明性能越稳定。

⚠️ 性能警报

长停顿警报

当GC停顿时间超过设定阈值时触发，通常表示需要调优GC参数。

频繁GC警报

当GC频率过高时触发，可能表示堆内存不足或分配速率过快。

内存泄漏警报

当堆利用率持续上升且回收效率下降时触发，可能存在内存泄漏。

吞吐量下降警报

当GC时间占比过高导致应用吞吐量下降时触发。

🔧 调优建议

堆大小调整

根据应用内存需求调整-Xms和-Xmx参数，避免堆过小导致频繁GC。

GC策略选择

根据应用特点选择合适的垃圾回收器，如延迟敏感选G1，吞吐量优先选Parallel。

新生代比例

调整新生代与老年代的比例，通常新生代占堆的1/3到1/4比较合适。

并发线程数

根据CPU核心数调整GC并发线程数，提高GC效率。
